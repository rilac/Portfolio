강의용 실습 프로젝트 (SPRING-LECTURE)

YouTube 강의 영상에서 사용되는 레퍼런스 구현체입니다. 학습자가 명세서대로 직접 만들어본 뒤 비교·참고할 수 있도록 가독성·구조 명확성을 최우선으로 작성했습니다.

도메인 분리 구조 (DDD-lite)

com.practice
├── domain
│   ├── hello   ← 워밍업 (가장 단순한 Controller-Service)
│   ├── member  ← 회원 CRUD (Entity-Repository-Service-Controller-DTO)
│   ├── post    ← 게시글 CRUD (인메모리 → JPA로 확장하는 시연용)
│   └── auth    ← 세션 기반 인증 (SessionManager로 직접 구현)
└── global
    ├── exception   ← CustomException + ErrorCode enum + GlobalExceptionHandler
    └── response    ← ErrorResponse 통일 포맷

의도적 설계 선택

  • PostEntity를 static Long AUTO_INCREMENT로 시작 → 이후 회차에서 JPA로 전환 — 점진적 학습 동선을 코드 자체에 반영했습니다.
  • SessionManager를 ConcurrentHashMap으로 직접 구현 — Spring Security를 도입하기 전, "세션이라는 게 결국 무엇인지"를 코드 레벨에서 이해하도록 유도했습니다.
  • ErrorCode enum을 도메인별 prefix(AUTH_001, POST-001)로 분리 — 에러 추적·문서화·확장이 쉬운 패턴을 처음부터 체화하게 했습니다.

깨달은 점

  1. "가르치는 것"이 가장 강력한 학습 도구임을 체감했습니다.

    강의 영상을 찍기 위해 같은 코드를 두세 번 다시 작성하고, "왜 이 어노테이션을 여기에 붙이는가"를 카메라 앞에서 설명하는 과정에서 막연히 알던 개념의 빈틈이 매번 드러났습니다. @Transactional이 작동하는 프록시 메커니즘, 세션이 결국 메모리 위의 Map이라는 사실, @RestControllerAdvice가 어느 시점에 동작하는지 — 모두 강의를 준비하면서 비로소 명확해진 것들이었습니다.

  2. 레퍼런스 코드의 품질 기준이 달라졌습니다.

    혼자 쓰는 코드와 20명이 따라 짤 코드는 요구되는 품질의 결이 다릅니다. 변수명 하나, 메서드 분리 하나, 패키지 구조 하나가 모두 학습자의 첫인상에 직결된다는 걸 체감하면서, 평소 작성하는 코드의 가독성·일관성·확장성에 대한 기준이 한 단계 올라갔습니다.
