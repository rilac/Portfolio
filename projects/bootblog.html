프로젝트 배경

사용 대상

블로그 서비스를 이용하는 일반 사용자. 회원가입 후 자신만의 블로그를 운영하고, 다른 사용자의 블로그를 방문해 글을 읽고 댓글을 남길 수 있습니다.

학습 배경

Spring Framework를 처음 학습하는 단계에서, 이론을 예제 수준에서 끝내지 않고 "사용자가 실제로 써본 적 있는 서비스 유형"을 직접 구현해 보자는 것이 출발점이었습니다. 티스토리·벨로그 같은 기존 블로그 플랫폼을 레퍼런스로 두고 핵심 기능을 최대한 비슷하게 재현하는 것을 목표로 했습니다.

구현 접근

Spring MVC 기본 요청 흐름 위에 JPA/Hibernate로 도메인 모델을 설계하고, HttpSession 기반 인증 + HandlerInterceptor로 보호 경로를 제어하는 표준적인 구조를 따랐습니다. 화면은 Thymeleaf + Bootstrap으로 구성해, 다중 테마(theme-1 ~ theme-8) 전환까지 구현해 완성도를 높였습니다.

학습 성과

회원가입·로그인, 글 CRUD, 댓글, 페이지네이션, 다른 사용자 블로그 탐색 등 블로그 플랫폼이 공통적으로 가진 기능을 한 번의 프로젝트 사이클로 경험. 이후 더 복잡한 Spring Boot 프로젝트(SSAFY BLIND)로 확장해 나갈 수 있는 기반을 마련했습니다.

주요 구현

  • ERD 설계 및 JPA 기반 테이블 생성 — Members, Board, Comments 엔티티 관계를 설계한 뒤 ddl-auto: create 설정으로 스키마를 자동 생성했습니다.
  • DTO 계층 설계 — LoginDTO, RegisterDTO, BoardDTO, CommentDTO, UpdateDTO 등 요청/응답을 분리한 구조로 Entity 직접 노출을 방지했습니다.
  • 댓글 기능 추가 구현 — Comments 엔티티와 Board 간 @ManyToOne 매핑, 작성/삭제 API와 게시글 상세 페이지 연동.
  • LoginCheckInterceptor 구현 — HandlerInterceptor를 구현해 WebConfig에 등록, 비로그인 사용자의 보호 경로 접근 차단 및 리다이렉트 처리.
  • 테스트 데이터 초기화 — TestDataInit으로 애플리케이션 구동 시 샘플 회원·글·댓글을 자동 주입해 반복 개발·검증을 효율화했습니다.

Project Management

PM 역할

일정 관리, 역할 분배, 코드 리뷰, 에러 리뷰를 담당했습니다. 5명으로 시작한 팀에서 한 명이 중도 이탈하는 상황을 만나, 남은 인원 간 업무 재분배와 일정 재조정을 주도했습니다.

깨달은 점

  1. DTO와 Entity 분리의 이유를 체감했습니다.

    처음에는 Entity 하나로 요청·응답·영속성까지 모두 처리해도 될 것 같았지만, 회원가입 시 노출되면 안 되는 필드·수정 시에만 오는 필드·응답에만 필요한 필드가 뒤섞이면서 금방 한계가 드러났습니다. 요청별 DTO를 분리한 뒤에야 Controller 코드가 단순해지고, 엔티티가 DB 관점에 집중할 수 있게 된다는 걸 실감했습니다.

  2. Interceptor는 "공통 관심사를 요청 처리 파이프라인에서 빼내는 도구"

    처음에는 각 Controller 메서드마다 세션 체크 코드를 if (session.getAttribute(...) == null) 식으로 반복해서 넣었다가, LoginCheckInterceptor로 추출하고 WebConfig에서 excludePathPatterns로 예외 경로만 지정하는 방식으로 바꾸니 컨트롤러가 훨씬 깔끔해졌습니다.

  3. PM 역할은 기술 이상으로 "예외 상황 대응"이 핵심

    5명으로 시작한 팀에서 한 명이 중도 이탈하면서 해당 기능의 분량을 나머지 인원이 나눠 맡아야 했습니다. 그때 가장 중요했던 건 남은 일정 대비 필수 기능과 포기 가능한 기능을 빠르게 재조정하는 일이었습니다. 완벽한 계획을 짜는 것보다, 변수가 생겼을 때 얼마나 빠르게 현실적인 재계획을 내놓을 수 있느냐가 PM의 실제 역량임을 처음으로 체감했습니다.
