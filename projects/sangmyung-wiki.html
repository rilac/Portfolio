핵심 구현

  • 학교 이메일 인증 기반 회원가입 플로우 — Google SMTP로 @sangmyung.kr 메일에 인증 요청을 발송하고, 요청·검증의 모든 단계를 DB에 로그로 남겨 재발송·만료·검증 상태를 추적하도록 설계. "재학생만 글을 쓸 수 있다"는 서비스 핵심 제약을 백엔드 레벨에서 강제했습니다.
  • 로그인 / 아이디 찾기 / 비밀번호 찾기·변경 API — 계정 식별 흐름 전반을 담당. 단계마다 필요한 검증 로직과 응답 포맷을 정의하고, 비밀번호 재설정 경로에는 BusinessException 기반 명확한 에러 반환을 적용했습니다.
  • 에러 코드 체계 확장 (CustomErrorCode enum) — 이메일 미존재·인증 실패·재전송 제한 등 인증 도메인 고유의 에러 케이스를 정의하고, @RestControllerAdvice 기반 GlobalExceptionHandler가 일관된 응답 포맷으로 내려주도록 검증 로직을 추가했습니다.
  • JWT 토큰 발급 로직 협업 — 다른 백엔드 멤버와 함께 JWT 기반 토큰 발급·검증 구조를 설계·구현. Spring Security의 SecurityFilterChain 위에 토큰 기반 인증을 얹고, 컨트롤러에서 @CurrentUser로 현재 사용자를 주입받을 수 있는 흐름까지 연결했습니다.

배운 점

"인증을 구현한다"는 한 문장 뒤에 실제로 얼마나 많은 상태와 예외가 숨어 있는지 체감했습니다. 메일 발송 실패, 토큰 만료, 재요청 제한 같은 케이스가 누락되면 서비스 신뢰가 한 번에 깨질 수 있다는 점을 알게 되었고, 그 이후로는 인증·예외 처리를 기능이 아닌 도메인으로 보고 설계하게 되었습니다.
