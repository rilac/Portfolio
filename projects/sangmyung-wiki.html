프로젝트 배경

사용 대상

상명대학교 재학생 전반. 가입 시 학교 이메일(@sangmyung.kr) 인증을 거쳐야 회원가입이 완료되는 구조로, 실제 재학생만 글을 작성·수정할 수 있도록 서비스 범위를 제한했습니다.

문제의식

학사 일정, 수강신청 팁, 교양 후기, 학식·셔틀·도서관 이용법 같은 정보가 한 곳에 정리되어 있지 않다는 문제에서 출발. 학우들은 매번 공지·카톡·커뮤니티 글을 뒤져야 했고, 학사 정보 하나를 찾는 데에도 시간이 걸렸습니다. "재학생이 직접 쌓고, 재학생이 직접 꺼내 쓰는" 학내 전용 지식 저장소가 필요하다고 판단했습니다.

솔루션

재학생 인증을 통과한 사용자만이 위키 문서를 작성·수정할 수 있도록 서비스 경계를 설정하고, 본인의 경험·팁·정보를 위키 페이지 형태로 축적할 수 있게 했습니다. 모든 문서는 가입자 누구나 열람 가능하며, 문서 수정 이력(DocLog)을 모두 기록해 이전 버전 추적과 기여 내역 관리가 가능하도록 설계했습니다. 자유로운 소통을 위한 커뮤니티 게시판을 함께 붙여 정형 위키와 비정형 대화가 공존하도록 구성했습니다.

기대 효과

학사·학교생활 관련 정보를 한 번에 찾을 수 있어 정보 탐색 시간 절약과 학내 생활 편의성 향상을 기대합니다. 정보가 축적될수록 후배 학번이 얻는 이득이 기하급수적으로 커지는 구조이기 때문에, 서비스가 오래 유지될수록 가치가 불어나는 플랫폼이 됩니다.

주요 구현

  • 학교 이메일 인증 기반 회원가입 플로우 — Google SMTP로 @sangmyung.kr 메일에 인증 요청을 발송하고, 요청·검증의 모든 단계를 DB에 로그로 남겨 재발송·만료·검증 상태를 추적하도록 설계. "재학생만 글을 쓸 수 있다"는 서비스 핵심 제약을 백엔드 레벨에서 강제했습니다.
  • 로그인 / 아이디 찾기 / 비밀번호 찾기·변경 API — 계정 식별 흐름 전반을 담당. 단계마다 필요한 검증 로직과 응답 포맷을 정의하고, 비밀번호 재설정 경로에는 BusinessException 기반 명확한 에러 반환을 적용했습니다.
  • 에러 코드 체계 확장 (CustomErrorCode enum) — 이메일 미존재·인증 실패·재전송 제한 등 인증 도메인 고유의 에러 케이스를 정의하고, @RestControllerAdvice 기반 GlobalExceptionHandler가 일관된 응답 포맷으로 내려주도록 검증 로직을 추가했습니다.
  • JWT 토큰 발급 로직 협업 — 다른 백엔드 멤버와 함께 JWT 기반 토큰 발급·검증 구조를 설계·구현. Spring Security의 SecurityFilterChain 위에 토큰 기반 인증을 얹고, 컨트롤러에서 @CurrentUser로 현재 사용자를 주입받을 수 있는 흐름까지 연결했습니다.

깨달은 점

  1. 외부 연동에서는 "상태 추적"이 기능만큼 중요합니다.

    처음 이메일 인증을 구현할 때는 "메일만 잘 가면 끝"이라고 생각했는데, 메일 전송 실패·만료된 인증 요청 같은 상황들이 서비스 체감 품질을 결정했습니다. DB에 인증 요청·검증 로그를 따로 남기기 시작하면서 디버깅이 쉬워졌고, 재전송 제한 같은 보조 정책도 이 로그 위에서 자연스럽게 붙일 수 있다는 걸 알게 됐습니다. 외부 연동(메일·결제·파일 업로드 등)을 다룰 때는 항상 "지금 상태가 어디쯤 와 있는가"를 DB에서 즉시 답할 수 있어야 한다는 원칙을 이 프로젝트에서 처음 세웠습니다.

  2. GlobalExceptionHandler + BusinessException + CustomErrorCode 3요소는 왜 하나의 세트인지 체감했습니다.

    인증 플로우에는 "이미 가입된 이메일", "미존재 계정", "인증 코드 불일치", "만료된 요청" 등 비슷해 보이지만 각기 다른 응답이 필요한 실패 케이스가 유난히 많습니다. 이 에러들을 컨트롤러 안에서 직접 처리하려니 코드가 순식간에 지저분해졌는데, CustomErrorCode enum에 한 줄만 추가하면 BusinessException 하나로 던지고 @RestControllerAdvice가 알아서 응답 포맷을 내려주는 구조로 바꾸면서 컨트롤러가 확 가벼워졌습니다.

  3. 세션 기반에서 JWT 기반 인증으로 넘어가는 경험은 단순한 기술 교체가 아니었습니다.

    React 프론트엔드가 백엔드와 분리되어 있어 상태를 서버가 들고 있을 수 없는 환경이었습니다. 왜 서버가 상태를 포기하고 토큰을 발급해야 하는지, Spring Security의 필터 체인이 왜 필요한지, 프론트가 매 요청마다 헤더에 토큰을 실어 보내야 하는지를 이 과정에서 비로소 납득했습니다. 직접 발급 로직 전부를 짜진 않았지만 팀원과 함께 설계에 참여하면서 "인증 방식의 선택은 아키텍처의 선택"이라는 걸 처음 깨달았습니다.
