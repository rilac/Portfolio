SSAFY BLIND

Mattermost API로 교육생 여부만 확인하고 자체 JWT를 발급해 완전 익명을 보장하는 SSAFY 교육생 전용 커뮤니티입니다. 기획부터 백엔드·프론트엔드까지 개인 프로젝트로 직접 설계·구현하고 있습니다.

프로젝트 배경

사용 대상

SSAFY(삼성 청년 SW 아카데미) 교육생 전반. Mattermost 계정을 보유한 현 교육생만 가입·로그인할 수 있습니다.

문제의식

교육 과정·취업·팀 갈등·평가에 대한 솔직한 고민을 털어놓을 공간이 없었습니다. 내부 공식 메신저인 Mattermost는 소속 반·이름이 모두 노출되고 사무국이 대화 내용을 열람할 수 있어 허심탄회한 글쓰기가 어렵고, 외부 커뮤니티는 SSAFY 교육생끼리만 통하는 맥락을 공유할 수 없습니다. "SSAFY 교육생인 건 확인하되, 누구인지는 알 수 없는" 소통 공간이 필요하다고 판단했습니다.

솔루션

Mattermost API를 인증 프록시로만 사용해 "현직 SSAFY 교육생 여부"만 확인한 뒤, 자체 JWT를 발급해 익명성을 완전히 보장하는 구조를 설계했습니다. 게시글·댓글의 author 관계는 DB에서만 유지하고 API 응답에서는 작성자 정보를 아예 제거해, 서버-클라이언트 사이 어디에서도 신원이 노출되지 않도록 했습니다. 신규 가입은 PENDING → ACTIVE 온보딩 플로우로 닉네임·부서를 설정하되, 이 정보 역시 게시물과 연결되지 않습니다.

기대 효과

교육 과정의 고민·팁·후기 등 실명에서는 꺼내기 어려운 정보가 자유롭게 축적되어, 재학 중인 교육생뿐 아니라 이후 기수 교육생에게도 살아있는 내부 정보원 역할을 할 수 있습니다. 완전한 익명성이 보장되므로 솔직한 피드백과 건설적인 토론이 가능해집니다.

주요 구현

인증 / 보안

  • Mattermost 인증 프록시 구현(MattermostClient) — RestTemplate로 SSAFY Mattermost의 /api/v4/users/login을 호출해 교육생 여부만 확인한 뒤 자체 JWT를 발급. Mattermost 세션은 저장하지 않으므로 서비스에는 "인증됨" 이상의 정보가 남지 않습니다.
  • JWT를 HttpOnly 쿠키로 관리 — CookieUtils 유틸리티로 JWT 쿠키 생성·삭제 로직을 일원화하고, httpOnly(true) + sameSite(Lax) 설정으로 XSS·CSRF 기본 방어를 적용했습니다.
  • JwtAuthFilter 설계 + shouldNotFilter() 크리티컬 수정 — OncePerRequestFilter를 상속해 쿠키에서 JWT를 추출·검증하는 필터를 구현. PENDING 상태 쿠키가 남아있을 때 /api/auth/login이 403으로 차단되는 버그를 shouldNotFilter()로 해결했습니다.
  • PENDING → ACTIVE 온보딩 플로우 — 신규 유저를 PENDING으로 생성한 뒤, 닉네임/부서 설정을 마치면 User.completeOnboarding() 도메인 메서드로 상태를 전환하고 ACTIVE JWT를 재발급. @Setter 대신 도메인 메서드로 상태 변경을 캡슐화했습니다.

게시글 / 댓글

  • 익명 게시글 CRUD — Post 엔티티에 author 연관관계는 유지하되, PostResponse.from()에서 작성자 정보를 의도적으로 제거해 API 응답 레벨에서 익명을 강제.
  • 조회수 벌크 UPDATE — @Query("UPDATE Post p SET p.viewCount = p.viewCount + 1 WHERE p.id = :id")로 DB 레벨 원자적 증가를 구현해 동시 요청 시 race condition을 방지했습니다.
  • N+1 해결 — 게시글 목록 조회 시 LEFT JOIN + COUNT JPQL 한 방 쿼리로 댓글 수까지 조회, 쿼리 수를 O(N) → O(1)로 감소.
  • Cascade 댓글 삭제 — @OneToMany(cascade = ALL, orphanRemoval = true)로 게시글 삭제 시 연관 댓글을 자동 삭제, FK 위반 방지.

예외 처리

  • @RestControllerAdvice 기반 전역 예외 핸들러 — InvalidCredentialsException(401), InvalidStateException(400), ForbiddenException(403), NoSuchElementException(404), MethodArgumentNotValidException(유효성 검사), 범용 Exception(500)을 각각 분리해 일관된 {"message": "..."} 응답 포맷으로 반환합니다.

프론트엔드

AI 페어 프로그래밍 도구(Claude)를 적극 활용해 골격을 빠르게 잡고, 인증 가드·페이지 흐름을 직접 검토·조정했습니다.

  • AuthContext + PrivateRoute 인증 가드 — 앱 마운트 시 GET /api/auth/me로 쿠키 유효성을 확인하고, 미인증 시 로그인 페이지로 리다이렉트. useAuth() 커스텀 훅으로 어디서든 인증 상태에 접근 가능.
  • 페이지 구성 — LoginPage → OnboardingPage → FeedPage → PostCreatePage → PostDetailPage 흐름으로 사용자 여정 전체를 설계·구현.

깨달은 점

  1. "익명을 보장한다"는 요구사항은 아키텍처 전체의 문제였습니다.

    단순히 화면에서 이름을 숨기는 것과 API 응답 자체에서 작성자 정보를 제거하는 것은 보안 수준이 완전히 다릅니다. Post.author를 DB에는 남기되 PostResponse.from()에서 매핑하지 않는 방식을 택한 것은 "DB 관리 편의"와 "API 레벨 익명성"을 모두 만족시키기 위한 설계 판단이었습니다. 이 과정에서 "어느 레이어까지 정보가 흘러야 하는가"를 기능 단위가 아닌 아키텍처 단위로 사고하는 습관이 생겼습니다.

개발이 계속 진행 중이며, 새로 깨달은 점은 단계별로 추가할 예정입니다.
